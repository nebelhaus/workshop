Once you're set up, day-to-day life with nebelhaus runs through one command:
**`haus`**. It wraps the Nix commands you'd otherwise type by hand вҖ” build,
switch, update, roll back вҖ” with safe defaults.

<Aside type="note" title="haus vs. bench">
The `haus` on your `PATH` (documented here) manages **your machine**. Family
*contributors* have a separate CLI, [`bench`](/internals/contributing/), in the
workshop repo for moving changes between repos вҖ” but as an end user you only
ever need `haus`.
</Aside>

## The everyday loop

Edit your config, then apply it:

```sh
haus edit       # open your host file in $EDITOR
# вҖҰmake changesвҖҰ
haus rebuild    # build first, then switch вҖ” a failed build never activates
```

`haus rebuild` always builds before switching, so a config with an error can't
take down your running system вҖ” you just see the build fail and fix it.

## Staying current

The rice lives upstream. To pull new versions:

```sh
haus update     # bump the nebelhaus pin, then rebuild
```

This updates the `flake.lock` in `~/.config/nix` to the latest nebelhaus and
rebuilds in one step. Because inputs are pinned, nothing changes until you run
this вҖ” your machine stays exactly as it was until you choose to move it.

## When something goes wrong

```sh
haus rollback      # atomically return to the previous generation
haus generations   # list generations you can roll back to
haus doctor        # health check: Nix, Xcode CLT, GUI agents
```

Every `switch` creates a new generation; `rollback` steps back to the last one
instantly. For macOS *settings* (not packages), the APFS snapshot the installer
took is your coarser rewind.

## The full command set

| Command | What it does |
|---|---|
| `haus rebuild` | Build, then switch вҖ” your everyday apply |
| `haus update` | Update the nebelhaus pin, then rebuild |
| `haus rollback [N]` | Return to the previous generation (or generation `N`) |
| `haus generations` | List available generations |
| `haus status` | Current generation and how stale the rice is |
| `haus edit` | Open your host file in `$EDITOR` |
| `haus doctor` | Diagnose Nix / CLT / agent health |

<Aside type="tip" title="From the palette, too">
[Pounce](/guides/pounce/) ships a **Rebuild** command вҖ” вҢҳSpace, type "rebuild" вҖ”
that runs `haus rebuild` in a floating terminal, plus a "Nix Config" command
that opens `~/.config/nix` in your editor. Handy when your hands are already on
the launcher.
</Aside>

## Catching up on another machine

If you rebuild from a second Mac (or after shipping changes from elsewhere),
just pull your config repo and rebuild:

```sh
cd ~/.config/nix && git pull
haus rebuild
```

Since everything is in git and pinned, the second machine converges to the exact
same system.

For the deeper picture of why pins work this way, see
[How the flakes fit together](/internals/flakes/).
